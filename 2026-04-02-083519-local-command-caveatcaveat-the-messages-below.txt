  handle and marks isPrepared = true. execute.cpp substitutes bound parameters
  (via substituteParams()) using type-aware quoting: strings get single-quote
  escaping, dates get DATE 'YYYY-MM-DD' syntax, timestamps get TIMESTAMP '...',
  booleans get TRUE/FALSE. Then posts to Trino via the existing REST client.

  application parameter descriptor), mirroring how SQLBindCol uses appRowDesc.
  Supports all common C types.

  4. SQLFetchScroll — SQL_FETCH_NEXT delegates to SQLFetch. All other
  orientations return SQL_ERROR with a clear message (Trino result sets are
  forward-only).

  5. Unicode W-suffix functions — New unicodeFunctions.cpp with 12 W-suffix
  entry points: SQLDriverConnectW, SQLExecDirectW, SQLPrepareW, SQLGetInfoW
  (string vs. numeric type detection), SQLColAttributeW, SQLDescribeColW,
  SQLGetDiagRecW, SQLTablesW, SQLColumnsW, SQLGetConnectAttrW,
  SQLSetConnectAttrW, SQLGetStmtAttrW, SQLSetStmtAttrW, SQLNativeSqlW. All use
  WideCharToMultiByte/MultiByteToWideChar for UTF-16 ↔ UTF-8 conversion.

⏺ Here's the full picture of your options, ranked by ease:

  ---
  Option 1: GitHub Actions (easiest, zero setup)

  The release workflow already builds on windows-2022 runners with MSVC + vcpkg.
   Add a simple build-on-push workflow to get compile feedback on every push
  without waiting for a tag:

  # .github/workflows/build.yml
  name: Build

  on: [push, pull_request]

  jobs:
    build:
      runs-on: windows-2022
      steps:
        - uses: actions/checkout@v4
        - uses: lukka/get-cmake@latest
        - uses: lukka/run-vcpkg@v11
        - name: Build x64
          shell: cmd
          run: |
            call "C:\Program Files\Microsoft Visual 
  Studio\2022\Enterprise\VC\Auxiliary\Build\vcvarsall.bat" x64
            cmake --preset x64-release
            cmake --build --preset x64-release

  Push to GitHub → watch the Actions tab. Free for public repos, and you get the
   exact same MSVC toolchain the release uses. No local setup at all.

  ---
  Option 2: Cross-compile with llvm-mingw on macOS (best local option)

  The driver's dependencies (curl, nlohmann-json, wincrypt, odbccp32) all have
  MinGW stubs, so this actually works:

  # Install the cross-compiler and vcpkg
  brew install llvm-mingw-w64
  brew install cmake ninja

  git clone https://github.com/microsoft/vcpkg ~/vcpkg
  ~/vcpkg/bootstrap-vcpkg.sh

  # Install deps for the x64-mingw-dynamic triplet
  ~/vcpkg/vcpkg install curl nlohmann-json gtest --triplet x64-mingw-dynamic

  # Build
  cmake -B build-mingw \
    -DCMAKE_TOOLCHAIN_FILE=~/vcpkg/scripts/buildsystems/vcpkg.cmake \
    -DVCPKG_TARGET_TRIPLET=x64-mingw-dynamic \
    -DCMAKE_SYSTEM_NAME=Windows \
    -DCMAKE_C_COMPILER=x86_64-w64-mingw32-gcc \
    -DCMAKE_CXX_COMPILER=x86_64-w64-mingw32-g++
  cmake --build build-mingw

  Produces a .dll you can copy to Windows. Caveat: doesn't run locally, but you
  get fast compile-error feedback without needing a VM or pushing to GitHub.

  ---
  Option 3: Windows VM on your Proxmox (best for full development)

  Since you already have Proxmox, this is the best environment for actual
  testing with PowerBI. A minimal setup:

  1. Spin up a Windows 11 or Windows Server 2022 VM
  2. Install Visual Studio Build Tools 2022 (choose "Desktop development with
  C++")
  3. Install Git and clone the repo
  4. vcpkg bootstraps automatically via lukka/run-vcpkg logic, or: git clone 
  https://github.com/microsoft/vcpkg && .\vcpkg\bootstrap-vcpkg.bat
  5. cmake --preset x64-release && cmake --build --preset x64-release

  This also lets you install the .msi, open PowerBI Desktop, and actually test
  the driver end-to-end against a Trino Docker container you run elsewhere on
  Proxmox.

  ---
  Option 4: UTM on your Mac (Windows on-device)

  UTM is free and runs Windows ARM natively on Apple Silicon (fast), or Windows
  x86 via emulation (slow). Good if you want Windows accessible locally without
  Proxmox. Same setup as Option 3 once Windows is running.

  ---
  My recommendation: Start with Option 1 (add the build workflow above, push,
  check for compile errors) since it costs nothing and gives you green/red
  feedback in 3 minutes. Use Option 3 (Proxmox VM) once you're ready to actually
   install and test the driver with PowerBI.
